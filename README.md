## Сканирование nmap

Команда для полного сканирования портов и определения версий служб:

```bash
nmap -sV -sC -p- 192.168.56.101
```

| Флаг | Описание |
|------|----------|
| `-sV` | определить версии служб |
| `-sC` | запустить стандартные скрипты (default scripts) |
| `-p-` | сканировать все 65535 портов |

## Обнаруженные уязвимости

### 1. vsftpd 2.3.4 — бэкдор (CVE-2011-2523)

**Служба:** FTP, порт 21  
**Описание:** В версии vsftpd 2.3.4 злоумышленник намеренно внедрил бэкдор: при отправке логина, содержащего символ `:)` (смайл), сервер открывает командный shell на порту 6200 без какой-либо аутентификации. Это позволяет получить полный удалённый доступ к системе.  
**Ссылка:** https://www.exploit-db.com/exploits/17491

---

### 2. Samba 3.0.20 — usermap_script (CVE-2007-2447)

**Служба:** SMB/NetBIOS, порты 139/445  
**Описание:** Уязвимость в параметре `username map script` файла `smb.conf`. При обработке имени пользователя через механизм подстановки shell-команд (`/bin/sh -c`) Samba не экранирует метасимволы, что позволяет выполнить произвольный код на сервере без аутентификации с правами демона Samba (обычно root).  
**Ссылка:** https://www.exploit-db.com/exploits/16320

---

### 3. UnrealIRCd 3.2.8.1 — бэкдор (CVE-2010-2075)

**Служба:** IRC, порт 6667  
**Описание:** В официальный дистрибутив UnrealIRCd 3.2.8.1 был внедрён бэкдор: при отправке строки, начинающейся с `AB`, сервер выполняет переданный код как системную команду от имени пользователя, под которым запущен демон. Позволяет получить удалённое выполнение кода без аутентификации.  
**Ссылка:** https://www.exploit-db.com/exploits/13853

## Вывод

Metasploitable 2 содержит намеренно уязвимые версии популярных сетевых служб, что делает её идеальной платформой для изучения методов анализа защищённости. Три описанные уязвимости относятся к классу **RCE (Remote Code Execution)** и эксплуатируются без аутентификации — наиболее критичный класс уязвимостей по шкале CVSS.

# Задание 2. Режимы сканирования nmap: SYN, FIN, Xmas, UDP

## Команды сканирования

```bash
# SYN-сканирование (полуоткрытое)
nmap -sS 192.168.56.101

# FIN-сканирование
nmap -sF 192.168.56.101

# Xmas-сканирование
nmap -sX 192.168.56.101

# UDP-сканирование
nmap -sU 192.168.56.101
```

---

## Чем отличаются режимы с точки зрения сетевого трафика

### SYN-сканирование (`-sS`) — «полуоткрытое»

Отправляется только первый пакет TCP-рукопожатия с флагом **SYN**. Полное соединение не устанавливается.

**Трафик в Wireshark:**
```
Клиент → Сервер:  TCP SYN  (порт X)
Сервер → Клиент:  TCP SYN-ACK  (порт открыт)  ИЛИ  TCP RST (порт закрыт)
Клиент → Сервер:  TCP RST  (соединение немедленно сбрасывается, не завершается)
```

- Быстрый и скрытный — сессия не попадает в логи приложений, т.к. соединение не завершается.
- Требует прав root/Administrator.

---

### FIN-сканирование (`-sF`)

Отправляется пакет с флагом **FIN** (обычно используется для завершения уже установленного соединения). Порт никогда не открывался, поэтому реакция сервера нестандартная.

**Трафик в Wireshark:**
```
Клиент → Сервер:  TCP FIN  (порт X)
Сервер → Клиент:  [нет ответа]  (порт открыт — RFC 793)
Сервер → Клиент:  TCP RST  (порт закрыт)
```

- Работает на основе особенности RFC 793: открытый порт обязан игнорировать пакет FIN без установленного соединения.
- Может обходить некоторые межсетевые экраны, которые фильтруют только SYN.
- На Windows результат ненадёжен — система отвечает RST на любой пакет, даже на открытый порт.

---

### Xmas-сканирование (`-sX`) — «рождественская ёлка»

Отправляется пакет одновременно с флагами **FIN + PSH + URG** — все биты «горят» как огни на ёлке.

**Трафик в Wireshark:**
```
Клиент → Сервер:  TCP FIN+PSH+URG  (порт X)
Сервер → Клиент:  [нет ответа]  (порт открыт)
Сервер → Клиент:  TCP RST  (порт закрыт)
```

- Логика определения открытости порта аналогична FIN-сканированию.
- Аномальная комбинация флагов легко выявляется IDS/IPS.
- Также не работает корректно на Windows.

---

### UDP-сканирование (`-sU`)

Работает с протоколом UDP (без установления соединения).

**Трафик в Wireshark:**
```
Клиент → Сервер:  UDP datagram  (порт X)
Сервер → Клиент:  UDP ответ  (порт открыт/открыт|фильтруется)
Сервер → Клиент:  ICMP Port Unreachable  (порт закрыт)
Сервер → Клиент:  [нет ответа / таймаут]  (порт открыт или фильтруется)
```

- Медленнее TCP-сканирований: ядро ОС ограничивает частоту ICMP-сообщений.
- Сложно отличить открытый порт от фильтрованного — оба могут молчать.

---

## Сводная таблица

| Режим | Флаги TCP | Открытый порт | Закрытый порт | Особенность |
|-------|-----------|---------------|---------------|-------------|
| SYN (`-sS`) | SYN | SYN-ACK | RST | Быстрый, скрытный, требует root |
| FIN (`-sF`) | FIN | нет ответа | RST | Обходит часть firewall, не работает на Windows |
| Xmas (`-sX`) | FIN+PSH+URG | нет ответа | RST | Аномальные флаги, заметен IDS |
| UDP (`-sU`) | — | UDP-ответ или тишина | ICMP unreachable | Медленный, сложно интерпретировать |

---

## Как отвечает сервер (Metasploitable / Linux)

| Ситуация | SYN | FIN | Xmas | UDP |
|----------|-----|-----|------|-----|
| Порт **открыт** | SYN-ACK | молчание | молчание | UDP-ответ или молчание |
| Порт **закрыт** | RST | RST | RST | ICMP Port Unreachable |
| Порт **фильтруется** | молчание или ICMP | молчание | молчание | молчание |

Metasploitable работает на Linux, поэтому FIN и Xmas-сканирования дают корректные результаты в соответствии с RFC 793: открытые порты молчат, закрытые отвечают RST. В Wireshark это хорошо видно: на открытые порты пакеты уходят, но ответа нет — nmap интерпретирует тишину как признак открытости.
